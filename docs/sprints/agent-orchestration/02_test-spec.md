## 2. テストシナリオ一覧

### 2-1. タスク作成・更新（UC-1 対応）

| ID | シナリオ | 種別 | 優先度 |
|----|----------|------|--------|

| TC-06 | `output`（ログ）を記録・更新できる | 正常系 | 中 |

| TC-08 | `to_agent` に存在しないエージェント名を指定するとエラーになる | 異常系 | 中 |
| TC-09 | `progress` に 101 以上を入力するとエラーになる | 異常系 | 中 |
| TC-10 | `progress` に負の数を入力するとエラーになる | 異常系 | 中 |
| TC-11 | 同一タスクを二重送信した場合、重複登録されない | 異常系 | 中 |

## 3. 手動テスト手順

### 事前準備

```
1. Supabase でテーブル（agent_tasks, agent_status）が作成済みであること
2. ダッシュボード URL にアクセス可能であること
3. テスト用ブラウザ: Chrome（PC）+ Safari/Chrome（スマホ）
4. 別タブ or 別デバイスでダッシュボードを開いておく（Realtime確認用）
```

---

### 手順 A: タスク作成・ステータス更新（TC-01〜11）

```
A-1. ダッシュボードを開く
A-2. 「タスク追加」ボタンを押す
A-3. 以下を入力して送信：
     - sprint_name: "agent-orchestration-test"
     - phase: "要件定義"
     - from_agent: "Tiara"
     - to_agent: "David"
     - task_type: "test"
     - title: "タスク作成テスト"
     - description: "手動テスト用サンプルタスク"

確認: タスク一覧に追加され、status = "pending" になっていること（TC-01, 02）

A-4. タスクのステータスを "running" に変更
確認: リスト表示が "running" に更新されること（TC-03）

A-5. ステータスを "done" に変更
確認: リスト表示が "done" に更新されること（TC-03）

A-6. 別のタスクを作成し、ステータスを "running" → "failed" に変更（TC-04）

A-7. title を空欄にして送信を試みる
確認: エラーメッセージが表示され、登録されないこと（TC-07）

A-8. progress を 101 に設定して送信を試みる
確認: エラーになること（TC-09）
```

---

### 手順 B: W字フェーズ遷移（TC-12〜23）

```
B-1. from_agent="David", to_agent="Tiara", phase="実装" でタスクを作成し status="done" にする
確認: フェーズ表示が「実装」→「テスト」に切り替わること（TC-15, 16）

B-2. from_agent="Tiara", to_agent="David", phase="テスト", task_type="bug_fix" でタスクを作成
確認: フェーズが「テスト」→「実装」に戻ること（TC-17）

B-3. from_agent="Tiara", to_agent="Imai", phase="テスト", task_type="deploy" でタスク作成
確認: Imai 宛タスクが正しく登録されること（TC-18）

B-4. 依頼先エージェントを busy 状態にした上でタスクを作成
確認: タスクが pending のまま待機し、処理が止まること（TC-22）

B-5. あるエージェントのタスクを failed にして、引き継ぎが発生しないことを確認（TC-23）
```

---

### 手順 C: ダッシュボード確認（TC-24〜30）

```
C-1. ダッシュボードトップを開く
確認: Conner, David, Tiara, Imai, Sakura の 5 名が表示されること（TC-24）

C-2. 各エージェントの status（idle/busy/offline）が表示されること（TC-25）

C-3. 現在のフェーズ（要件定義/実装/テスト/リリース/振り返り）が表示されること（TC-26）

C-4. タスク履歴ページを開き、過去のタスクが一覧できること（TC-27）

C-5. タスクが 0 件の状態でダッシュボードを開く
確認: エラーなく「タスクなし」等の空状態が表示されること（TC-29）

C-6. Supabase URL を意図的に間違えて読み込む（または Supabase を一時停止）
確認: 「接続エラー」等のユーザー向けメッセージが表示されること（TC-30）
```

---

### 手順 D: リアルタイム同期（TC-31〜35）

```
D-1. PC と スマホ（or 別タブ）の両方でダッシュボードを開く

D-2. PC 側でタスクのステータスを更新する
確認: スマホ側でページ再読み込みなしに反映されること（TC-31, 32）

D-3. PC 側でエージェントステータスを idle → busy に変更
確認: スマホ側に即時反映されること（TC-33）

D-4. スマホの Wi-Fi を一時的にオフにし、再接続する
確認: 再接続後に最新状態が表示されること（TC-34）
```

---

### 手順 E: レスポンシブ確認（TC-36〜39）

```
E-1. Chrome DevTools で以下の幅に切り替えて表示を確認：
     - 375px（iPhone SE 相当）→ 崩れなし（TC-36）
     - 768px（タブレット相当）→ 崩れなし（TC-37）
     - 1280px（PC）         → 崩れなし（TC-38）

E-2. スマホ実機（または 375px エミュレーション）でタスク作成を操作
確認: ボタン・フォームが正常に動作すること（TC-39）
```

## 4. 合否判定基準

| 判定 | 条件 |
|------|------|
| Pass | 期待結果と一致する |
| Fail | 期待結果と異なる / エラーが発生する |
| Skip | 実装未完のため検証不可（理由を記録） |

**Go 判定**: 優先度「高」の全シナリオが Pass であること。

## 6. 既知リスク・注意事項

- Supabase Realtime は WebSocket を使用するため、企業ネットワーク等でブロックされる場合がある
- スマホ実機テストは Safari（iOS）と Chrome（Android）で動作差異が出る可能性がある
- `failed` 状態からの自動引き継ぎ停止は、フロントエンド実装方針によってテスト手順が変わる可能性がある（David に確認要）
