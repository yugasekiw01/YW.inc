## 総評

方向性は正しい。W字モデルの自動フロー化という核心価値は明確で、Davidのドラフトはよく整理されている。ただし **MVP のスコープが広すぎる**点と、**ユースケースに重要な抜け**がある点を指摘する。

---

## 1. スコープの妥当性

### 問題: 「エージェントの自動起動」はMVPに重すぎる

現在のMVPに含まれている「エージェントの自動起動（タスク受信→自動で実行開始）」は、このシステムの最大の技術的難所であり、MVP段階での実装は **リスクが高い**。

**根拠:**
- Claude Code は常駐プロセスではなく、都度起動するセッション型のツール。「自動起動」の仕組みが未設計（Webhook？ポーリング？Supabase Realtime subscription?）
- 自動起動の実装が遅延した場合、ダッシュボード完成後に長期間「手動起動」のまま運用するリスクがある
- 逆に、ダッシュボード＋手動タスク管理だけでも **現状の「口頭での手動伝達」より大幅に改善される**

### 推奨: MVPを2段階に分割する

| フェーズ | 内容 | 価値 |
|----------|------|------|
| **MVP-A（今回）** | タスク管理テーブル + API + ダッシュボードUI + Realtime | 可視化・手動オーケストレーション |
| **MVP-B（次スプリント）** | エージェント自動起動の仕組み | 真の自動化 |

MVP-Aだけでもスプリントの価値として十分成立する。自動起動は「どのプロセスがSupabaseを監視するか」の設計が固まってから着手すべき。

## 2. 受け入れ条件の追加/修正

### 追加が必要な必須条件

```
- [ ] タスクをキャンセルできる（実行前・実行中ともに）
- [ ] エラー時（status=failed）にエラーメッセージが記録・表示される
- [ ] 同じエージェントに複数タスクが届いた場合、キュー順に処理される
```

### 追加が必要な推奨条件

```
- [ ] タスクの優先度設定（urgent/normal）
- [ ] 各タスクに紐づくスプリント名でフィルタリングできる
```

### 修正提案

**既存の「推奨」→「必須」に昇格すべき項目:**
- `W字モデルのフェーズ表示` → 必須へ。これがないとダッシュボードとしての核心価値が失われる

## 3. ユースケースの抜け漏れ

### 抜けているUC

#### UC-4: タスクのキャンセル・中断
```
1. ユーザーまたはエージェントが実行中タスクをキャンセル
2. status が running → cancelled に更新
3. 後続エージェントへの引き継ぎは発生しない
4. ダッシュボードに「キャンセル済み」として表示
```

#### UC-5: タスク失敗時のエスカレーション
```
1. エージェントがタスクを failed に更新（エラーメッセージ付き）
2. ダッシュボードにアラート表示
3. ユーザーが「リトライ」または「スキップ」を選択
4. リトライ: status を pending に戻し、再実行
```
> 現状のドラフトにはバグ発見→David差し戻しのUCはあるが、**タスク自体が失敗した場合**のフローが未定義。

#### UC-6: Sakuraによるスプリント起動
```
1. ユーザーがSakuraに新スプリント開始を依頼
2. Sakura が sprint_name を設定し、Phase 1 タスクを各エージェント（Conner/Tiara/Imai）に一括作成
3. 各エージェントが並行してPhase 1を開始
```
> UC-2のW字フローでSakuraは終端（振り返り完了報告）にしか登場しないが、**起点（スプリント開始の指揮）** も Sakura の役割のはず。明示化が必要。

### スコープ外だが将来必要になる論点

- **並行タスクの衝突**: 同じエージェントが2スプリントから同時に依頼を受けた場合（「今回やらないこと」に追記推奨）
- **引き継ぎ時の添付ファイル**: W字の各フェーズで仕様書・テスト結果などを次エージェントに渡す仕組み。`description`フィールドだけで足りるか要確認

## 4. データ設計へのコメント

### `agent_status` テーブルの問題

`status: idle/busy/offline` の `offline` 判定が曖昧。Claude Codeは常駐しないため、`last_seen`が一定時間更新されなければofflineと見なすのか、それとも別の仕組みか。

**提案:** `status` の定義を以下に絞り込む

| status | 意味 |
|--------|------|
| `idle` | タスクなし / 待機中 |
| `busy` | タスク実行中 |
| `stale` | last_seen から X 分以上無応答（要定義） |

### `agent_tasks` テーブルへの追加提案

| カラム | 型 | 理由 |
|--------|-----|------|
| `priority` | text (urgent/normal) | 緊急バグ修正等の優先制御 |
| `status` への `cancelled` 追加 | - | UC-4 対応 |
| `error_message` | text | 失敗理由の記録 |
