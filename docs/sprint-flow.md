# YW.inc 機能スプリント フロー

## 開発モデル

アジャイル × W モデル（シフトレフト型 V 字）。  
1機能 = 1スプリント。全エージェントが要件フェーズから並走する。

```
要件定義
  ├── Conner  : 仕様・受け入れ条件の定義
  ├── Tiara   : テスト仕様・完成基準を同時に書く
  └── Imai    : インフラ・デプロイ要件を確認

実装
  └── David   : Tiara の完成基準を見ながら開発

テスト
  └── Tiara   : 自分で書いたテスト仕様で検証

リリース・運用
  └── Imai    : 最初から把握済みなので手戻りなし

振り返り
  └── Conner  : 仕様とのズレ・次スプリントへの改善点
```

---

## フェーズ別 チェックリスト

### フェーズ 1：要件定義（Conner + Tiara + Imai 並走）

**Conner が定義するもの**
- [ ] 機能の目的（誰が、何を、なぜ）
- [ ] スコープ（今回やること / やらないこと）
- [ ] 受け入れ条件（完成の定義）

**Tiara が定義するもの**（要件定義と同時）
- [ ] テストシナリオ一覧（正常系 / 異常系）
- [ ] 手動テスト手順 or 自動テストの方針

**Imai が確認するもの**（要件定義と同時）
- [ ] 新規インフラが必要か（DB、API、ストレージなど）
- [ ] 監視・ログの要件
- [ ] デプロイ手順の変更有無

---

### フェーズ 2：実装（David）

- [ ] Conner の仕様書を参照
- [ ] Tiara のテストシナリオを完成基準として使う
- [ ] 実装完了後、Tiara にレビュー依頼

---

### フェーズ 3：テスト（Tiara）

- [ ] フェーズ 1 で定義したテスト仕様を実行
- [ ] バグは David にフィードバック
- [ ] 完成基準をすべて満たしたら承認

---

### フェーズ 4：リリース・運用設定（Imai）

- [ ] フェーズ 1 で確認したインフラ変更を適用
- [ ] デプロイ実施
- [ ] 監視・アラートの確認

---

### フェーズ 5：振り返り（Conner）

- [ ] 仕様とのズレがあったか
- [ ] 次スプリントに持ち越す課題
- [ ] プロセス改善点

---

## Sakura（秘書）の役割

各スプリント開始時に以下を各エージェントに伝える：

```
機能名：〇〇
目的：〇〇
今日の担当フェーズ：〇〇
参照ドキュメント：docs/sprint-flow.md
```

スプリント進行中は各エージェントの進捗を管理し、
次フェーズへの引き継ぎを調整する。

|---|---|---|
